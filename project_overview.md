# Project Spec: Native C++ Hexagon NPU Inference Engine

## 1. Executive Summary
**Objective:** To design, train, and execute a static, custom statistical machine learning model (such as a tabular regression network for player performance prediction) natively on the Snapdragon X Elite Hexagon Tensor Processor (HTP). 
**Methodology:** Bypass standard high-level runtimes (PyTorch, ONNX Runtime) by manually extracting model weights and constructing the mathematical execution graph from scratch in C++ using the Qualcomm AI Engine Direct SDK (QNN).
**Hardware Target:** Snapdragon X Elite (X1E-78-100) / 45 TOPS Hexagon NPU.

## 2. Technology Stack
* **Modeling & Training:** Python 3.12 (ARM-native), PyTorch, NumPy.
* **Low-Level Execution:** C++, Qualcomm AI Engine Direct SDK (QNN API).
* **Build Tools:** Visual Studio 2022 (v17.4+), MSVC ARM64EC toolchain.
* **Debugging Aids:** ONNX Runtime (QNN Execution Provider).

---

## 3. Phase 1: Python Training & Precision Preparation
This phase focuses on training a statistically rigorous model while adhering strictly to hardware-imposed constraints.

### Architectural Constraints
* **Topology:** A fixed-shape feed-forward network (MLP). The standard flow must be strictly `input -> Gemm (Fully Connected) -> ReLU -> Gemm -> output`.
* **Static Shapes:** The HTP path explicitly does not support dynamic shapes. The input vector size (e.g., the exact number of historical statistical features fed into the model) must be hardcoded and static.
* **Standard Operations:** Confine the network to natively supported HTP operations: `Add`, `Relu`, `Gemm`, and `MatMul`. Custom activations require a complex, separate op-package generation workflow and must be avoided.

### Weight Extraction & Quantization Prep
* Train the model to convergence in PyTorch.
* Extract the weights (`state_dict`) into a raw, structurally simple format (JSON, CSV, or raw binary) that the C++ engine can easily parse.
* **Precision Expectation:** While PyTorch operates in FP32, the Hexagon NPU prefers reduced or mixed precision (INT8, INT16, or FP16). The exported weights must be ready for downcasting, and validation will require tolerance-based numerical agreement rather than exact bit-for-bit identity.

---

## 4. Phase 2: C++ QNN Graph Construction (The Core Engine)
This is the low-level systems programming phase where the mathematical graph is manually authored.

### Initialization & Memory Management
* Initialize the QNN HTP backend using the provided Windows ARM64 DLLs (`QnnHtp.dll`).
* Allocate memory buffers for the network inputs, weights, and outputs. Utilize Qualcomm's shared-buffer support to eliminate redundant data copies between the CPU and NPU memory domains.

### Graph Authoring
* Read the extracted PyTorch weights into the allocated C++ buffers.
* Utilize the QNN composable graph API to build the execution logic node-by-node.
* Use `QnnGraph_addNode` to define the configuration, inputs, and outputs for every single tensor operation (e.g., mapping a `Gemm` operation to the specific memory addresses of your extracted weights).

### The "Cheat Code" (Reference Generation)
* Before finalizing the C++ graph, pass the PyTorch model through ONNX Runtime’s QNN Execution Provider.
* Utilize ORT-QNN's debugging tools to dump the autogenerated QNN graphs as JSON. Use this JSON as a structural blueprint to ensure your manual C++ graph correctly maps the node connections.

---

## 5. Phase 3: Validation, Execution & Benchmarking
Execution must be systematically validated to isolate precision errors from graph architecture errors.

### The Debugging Ladder
Do not target the NPU immediately. Validate the mathematical outputs at each step before moving down the stack:
1.  **PyTorch CPU Baseline:** Establish the FP32 mathematical ground truth.
2.  **Custom C++ CPU Reference:** Verify your C++ weight-loading and basic matrix math.
3.  **QNN CPU Backend (`QnnCpu.dll`):** Validate that your QNN graph API calls are structurally correct without introducing NPU quantization noise.
4.  **QNN HTP Backend (`QnnHtp.dll`):** Execute on the Hexagon NPU, applying tolerance-based checks to account for FP16/INT8 precision shifts.

### Benchmarking Methodology
* **Cold Start vs. Warm Inference:** NPU performance profiling must strictly separate startup overhead from calculation time. The HTP requires graph-finalization and context-binary creation, which introduces significant fixed overhead. 
* **Measurement:** Implement high-precision C++ timers to log the cold start (graph compilation) separately from the hot loop (steady-state inference). On a highly compact tabular MLP, NPU startup latency may eclipse CPU calculation time; the performance win will be demonstrated in the warm inference loop.
